3. No model-generated Android plan was available to score for quality, correctness, or accuracy.
4. The API /health endpoint reported status ok while generation was not producing responses within practical client timeouts.
5. The vLLM /v1/models endpoint reported the model as available while generation still timed out.
6. The live API route is configured with fallback_backend none, so generation failure yields no deterministic backup response.
7. The client received curl HTTP code 000 rather than a structured planner error body within the bounded timeout.
8. The configured server-side planner timeout is much longer than an interactive assistant can reasonably wait.
9. A single stuck generation is enough to make a 200-prompt live evaluation unsafe because requests may queue behind the busy worker.
10. The current health checks do not include an active generation probe, so they miss the most important readiness failure.
11. The low, medium, high, and xhigh routes all report the same vLLM backend with no custom route configuration, so mode-specific behavior could not be verified during this run.
12. No schema-valid TaskPlan JSON was produced by the model during the live attempt.
13. No implementation_tasks, acceptance_checks, risks, or questions_for_user could be inspected from the live model response.
14. No latency_ms field from the API response was available because the API did not complete the request.
15. The direct vLLM and API behavior suggest backend liveness is weaker than endpoint availability.
16. The evaluation could not safely continue to the remaining 199 prompts without risking a large backlog of long-running model calls.
